# 技術用語 → 日常語（自分用）

**このファイルの役割**: 用語の日常語化・たとえ・**初出の扱い・英語名の書き方・用語解説ボックスの定型**の SSoT。ワークフロー上は [SKILL.md](../SKILL.md) Phase 2 で表を作り、Phase 6 で HTML に反映する。**1回につき1テーマだけ**・**「今日の3つの要点」の3行の意味**・**まとめの必須3点**は [SKILL.md](../SKILL.md) の該当節のみを正とする（ここでは繰り返さない）。

## 変換の原則

1. **まず「何をするか」で名付ける**: 本文では日常語を主にする。**英語の固有名詞**を出すときは、括弧で正式名を並記するか、**直後に「つまり〜」で日常語に言い切る**（どちらか必ず）。
2. **たとえは用語1つあたり1つ以上**: 講座・予約・セミナー準備など、自分の生活・仕事に寄せる。※ページ全体で「たとえ話を何セクション置くか」は [SKILL.md](../SKILL.md) Phase 3（**たとえ話を3つ以上**）の対象で、こことは別スコープ。
3. **初出の扱い**: その用語を本文に**初めて出す**ときは、**用語解説ボックス**で止める（インラインだけの初出はしない）。用語を出さず言い換えだけに済ませるならボックス不要（品質の最終確認は [exemplar.md](exemplar.md) のチェックリスト）。用語解説ボックスでは **「この図解では〇〇と呼ぶ」** を必ず書く。

## よく使う変換表（AI・ドキュメント系）

### ドキュメント・開発ツール周り

| 技術用語 | 日常語 | たとえ |
|---------|--------|--------|
| Skill（Claude 等） | 手順書の束 | 店のマニュアルを何冊も束ねたもの。状況に合わせて開く本が変わる |
| Markdown | 記号つきの文書形式 | 見出しや箇条書きが書きやすいメモの書き方 |
| Repository / リポジトリ | プロジェクトのフォルダ一式 | 引き出しに資料を入れて共有する箱 |
| CLI / コマンド | 文字で指示を出す画面 | 電話でオペレーターに話す代わりに、短い合言葉で頼む |
| Hook | 割り込みの決まり | 施術の前に必ず行う問診のような「入る前のチェック」 |

### プログラミング基礎

| 技術用語 | 日常語 | たとえ |
|---------|--------|--------|
| API | 窓口 | 決まった手続きで、欲しい結果をもらう場所 |
| パラメータ | 渡す条件 | 予約フォームの入力項目 |
| コールバック | あとで呼ぶ処理 | 「終わったら連絡して」と預ける約束 |

### AI

| 技術用語 | 日常語 | たとえ |
|---------|--------|--------|
| プロンプト | AIへの依頼文 | 依頼書・レシピの材料リスト |
| コンテキスト | これまでの前提 | 続きの会話・渡した資料の束 |
| エージェント | 自律して動くAI | 指示だけで動く秘書 |
| ツール（AIが使う） | AIが触れる道具 | 秘書が使える電話・カレンダー・ファイル |

### AIリスク・安全（パーソナル図解・リスク整理の回）

| 技術用語 | 日常語 | たとえ |
|---------|--------|--------|
| ハルシネーション（幻覚） | でたらめに見える正しい返答 | 自信満々なのに地図が古い道案内 |
| バイアス（偏り） | 学習データの偏りが出た答え | アンケートが一部の人ばかりだと結論が傾く |
| ディープフェイク | にせ音声・にせ動画 | そっくりな人形劇。本人の許可とは別問題 |
| プロンプトインジェクション | 悪意のある指示のすり込み | 貼り紙に「裏の指示も読んで」と小さく書かれている |
| 個人情報の再識別 | 匿名のつもりが、組み合わせでバレる | 苗字だけ→苗字＋勤め先で特定されうる |
| ガバナンス | ルールと責任の決め方 | 教室の「提出期限と採点の決まり」に近い |
