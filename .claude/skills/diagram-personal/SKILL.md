# Diagram Personal（パーソナル図解）

## 起動のしかた

**トリガーは「開始」だけ**（この文字列だけ、または「開始」が依頼の核になっている場合）。それ以外の入力では本スキルを起動しない。

ユーザーが「開始」したら **確認を挟まず** **Phase 0 から Phase 9 まで一気通貫**で進める。題材は **毎回1テーマだけ**選ぶ（下記「リサーチの固定」）。ユーザーがチャットで特定の用語・フレーム名・論点を指定した場合はそれを優先し、無指定なら Phase 0 で **新規事業を始める人がつまずきやすい論点**、または **制度・支援・市場の動きが絡み実行時点で調べる価値が高い論点**から **1テーマ** を選ぶ。

図解 HTML を生成し **surge.sh にデプロイする**（Phase 9 は省略しない）。

### リサーチの固定（「開始」時の SSoT）

- **1回の「開始」＝1枚の図解＝新規事業構築に効く「ビジネス基礎知識」1テーマの深掘り**（考え方・用語・フレームのいずれかに絞る）。百科や「起業マニュアル全部入り」の一覧にしない。複数テーマを並べて浅く触れない。
- **最新性**: 法制度・支援制度・税制・助成、市場の話題は実行日時点で変わりうるため、**必ずその実行日時点で `WebSearch`（および必要なら公的ページの取得）を行い**、根拠にする情報の**公表日・改定日**が分かるようにメモする。古典的な経営・事業フレームは、**原著・公的ガイド・大学公開資料・信頼できるビジネスメディア**などで補い、「いつの説明か」を分かるように書く。古い記事だけで「今こうなる」と断定しない。出典一覧には**参照した主URL**と、分かれば**情報の日付**を併記する。
- **情報源の優先順位**: 官公庁・独立行政法人（創業支援・中小企業支援）・業界団体の公式、一次に近い解説、主要メディアの取材記事など。信頼できるガイド・教科書的整理の出典を優先する。**SNSの断片だけで断定しない**。
- **偏り**: 毎回の整理は入手情報に依存するため**網羅的な学術調査ではない**。冒頭リードに **「今回取り上げる1テーマは〇〇。選んだ理由は（1行）」** と書く。
- **深掘りで埋める観点**（図解で一目にする）:
  - **何の考え方・用語か**（日常語の名前＋一言定義）
  - **なぜ事業を作る人がここでつまずきやすいか**／**他の基本とどうつながるか**
  - **自分の事業にどう効くか**（いまの個人事業・これからの顧問・コンサル事業）と、**世の中・ビジネスではどう位置づけられるか**
  - **今日からの一歩**（チェック1つ、試すこと、調べる先・相談窓口のイメージ）
  - **出典**（URL と分かる範囲の日付）

## 読者・仕事・ゴール・「今日の3つの要点」の意味（本スキル内の正）

**読者・仕事・成功条件・事業構築向け図解の解釈の正はこの節だけ**とする。他ファイルでは**同一内容を別の言い回しで長く書かない**（`exemplar.md` の `diagram-maji` 比較は**対照用の短い表**に限り、各セルは箇条書きレベルに留め、**読者・成功の定義本文は繰り返さず**本節や各リファレンスへのリンクに寄せる。チェックリストは本節へのリンクで足りる）。Phase 7 のレビュー用ペルソナは**この節と一致**させる（プロンプト内に全文を繰り返さず、実行時にこの節を読む前提にする）。

### 読者

- **読者は自分ひとり**。
- **新規事業の構築は初めて**で、**事業を作るうえでの基本的な考え方はほぼわからない**。**ビジネス用語もほぼわからない**前提で書く（**前提知識ゼロの中学生でも通じる**くらいに噛み砕く）。
- AI講師としてデジタルツールには触れているが、**経営・事業設計の用語は別物**として当面は知らないつもりで説明する。長文や専門用語の連続で疲れやすい。

### 仕事・生活の文脈（セリフやたとえの軸）

- **いま**: 個人事業主として**AI講師**。集客から商品設計・セミナー登壇まで**自ら担当**。
- **今後**: **個人事業主向けのAI顧問・コンサル事業**を立ち上げ、展開していく予定。
- **今後（自分のAI顧問・コンサル事業の想定）**:
  - **ターゲット**: **個人事業主**。
  - **立ち位置**: 各クライアントの業務に**直接入り込み**、業務フローを再構築して **AI に最適化した業務形態**にする。**クライアントのAI部門として**入り込むイメージ。
  - **届け方の中心**: **オンラインセミナーが主役ではない**。**実際の業務フロー全体**を対象に最適化する。
  - **ツール**: 既存ツールの利用に加え、**個人に最適化した自動化ツールの開発**も行う。
  - **期間・価格（検討中）**: 伴奏はおおむね **3ヶ月**。価格は **15〜30万円**のレンジで検討中（図解内で金額に触れる場合は税込／税抜を分かる範囲で統一）。

たとえ話は **講座・集客・商品設計・個人事業・これからの顧問事業・業務フロー** になじむものを優先してよい（無理に入れない）。

### 典型ニーズ（優先度設計の参考）

（ヒアリングより）Geminiの予約アクションなどで**「新規事業を作るにあたって抑えておくべきビジネスの基本的な考え方」**の文章を出してもらったが、**文章が多すぎ**、**専門用語の羅列**で理解不能だった。**前提知識がない中学生でも理解できる**ように噛み砕いた図解があればよかった、という場面を想定する。**新規事業を構築・展開していくうえで必須になりうる基礎**が**一目で**わかるように取捨選択する。**専門用語は日常語に翻訳**する。

### 成功の定義（ゴール）

読み終わったあとに次を満たすこと（レビュー・チェックリストもこの定義に揃える）。

1. **今日取り上げた1テーマ**を、**自分の言葉で30秒程度**に説明できる。**何を指すか・自分のこれからの事業とどう関係するか・次に何を試せるか**が腹落ちしている。
2. **「今日の3つの要点」**（下表）がそれぞれ一言で言え、**長文の調査レポートを読まなくてよい圧縮**になっていること。
3. **今後、新規事業を構築し展開していく**自分が、図解の内容を**事業構築の判断・行動に活かせる**イメージが持てていること（図解上で「自分用の一行」「次の一歩」が添えられている）。

### 「今日の3つの要点」とセクション見出し

ページ上の **「まず押さえる3つ」相当のセクション**の **h2 見出し文言は「今日の3つの要点」に統一する**（旧称「この競合でまず押さえる3つ」「まず覚える3つ」「まず押さえる3つ」は使わない）。中身は**毎回の1テーマに紐づける**（仕事・社会・講師の「固定3側面」に必ずしも対応させない）。

| 行 | 内容（各行に一言要約＋補足。必要なら自分の事業での使い道・次の一歩のヒント） |
|----|----------------------------------------------------------------|
| ① | **この考え方（・用語）の正体**（何か／事業づくりでなぜ出てくるか、日常語） |
| ② | **自分の事業に近い意味**と、**世の中・ビジネスでの位置づけ**（短く二列イメージでよい） |
| ③ | **今日からの使い方**（チェック1つ、試すこと、調べる先・相談の目安など） |

**全体の地図**: 1テーマ深掘り用に、**「定義 → 事業での現れ方（具体例）→ 使い方・次の一歩」**の流れが一目でわかる**小さな地図（3ブロック以内）**にする。ラベルはテーマに合わせて毎回変える。

**本編**: **その1テーマについて**、対話＋本文で **3〜5セクション**程度まで深掘りしてよい（見出しはテーマ依存。`id` は `section-*` で統一）。

### まとめ（最終セクションの必須構成）

**まとめの対話または本文の直後**に、次の**3点を必ず明示**する（箇条書きでも短い段落でもよい）。

1. **今日取り上げたテーマについて一言でまとめる**（リード文レベルの短さ）。
2. **自分の事業（いまの個人事業・これからの顧問・コンサル事業）の視点**と、**より広いビジネス文脈**のそれぞれで、**どう効くか・どう位置づけられるか**（各1〜3文程度）。
3. **新規事業を構築・展開するときに外さないポイント**（2〜5項目程度）。

学習者の最終吹き出しでは、上記（1）〜（3）を**自分の言葉に言い換えた要約**＋「つまり合ってますか？」の確認ができるとよい（[dialogue-personas.md](references/dialogue-personas.md) のまとめの型に沿う）。

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**explore サブエージェント**で、本スキル冒頭の **「リサーチの固定」** に従い、**新規事業構築に効くビジネス基礎知識1テーマ**について実行時点の公開情報を集め、**事業づくり初心者の自分**が行動に移せる材料を揃える。

```
Task({
  subagent_type: "explore",
  description: "事業構築・ビジネス基礎1テーマ深掘り用リサーチ",
  prompt: `
    次のミッションで調査してください（競合企業・個人サービスの比較・価格調査はスコープ外）。

    【ミッション】
    実行日時点で **WebSearch を必ず使い**、**新規事業を作るうえで押さえておくとよいビジネス基礎**（考え方・用語・フレームのいずれか）を**1テーマだけ**選んで深掘りするための材料を集める。信頼できる公開情報（官公庁・独立行政法人の創業・中小支援、公的ガイド、業界団体、一次に近い解説、主要メディアなど）を優先する。

    【テーマの決め方】
    - ユーザーがチャットで特定の用語・フレーム名・論点を指定していればそれを採用する。
    - 無指定なら「新規事業者がつまずきやすい」「制度・支援・市場の動きで意味が変わりうる」論点から**1つ**選び、選定理由を1行で書く。

    【調査項目】（箇条書きで返す）
    1. **今日の1テーマの名称**（日常語）と**1行定義**
    2. **なぜ今それを取り上げるか**（制度・支援の更新、よくある誤解、実行時点の論点など、日付つきで）
    3. **自分の事業に近い意味**（個人事業・これからの顧問・コンサル事業に結びつく具体例2つまで）
    4. **世の中・ビジネスでの位置づけ**（具体例2つまで）
    5. **事業構築で外さない核**（箇条書き3〜5）
    6. **自分用の短い一行**（メモ・判断のときに使う文案、コピペ用に2案まで）
    7. **参照URL**（一次・準一次優先。可能なら各リンクの公表年または更新日）

    結果は箇条書きで返してください。
  `
})
```

### Phase 1: ソース読み込み

1. Phase 0 で得た URL を `WebFetch` で取得し、足りなければ `WebSearch` 等で補う
2. **情報の全体像**（見出しの地図）を先に把握する
3. Phase 0 の整理と照合し、抜け・誤りを埋める
4. **鮮度チェック**: 主要な根拠URLの日付を確認し、古い情報だけで「今こうなる」と断定している箇所があれば、Phase 0 を補足調査するか、本文で「〇年時点の情報」と明示する

### Phase 2: 用語翻訳表を作る

技術用語・業界用語を洗い出し、日常語の名前とたとえを付ける。**変換の原則・初出・英語名の書き方・用語解説ボックスの定型・HTML例**はすべて [references/term-dictionary.md](references/term-dictionary.md) に従う（ここでは繰り返さない）。

### Phase 3: たとえ話を3つ以上

最終的なページ上のセクション順・塊の切り方は、[references/exemplar.md](references/exemplar.md) の「うまくいく構成」を SSoT とし、Phase 3〜6 でそこへ落とし込む。

| 種類 | 例 |
|-----|-----|
| 身近 | 集客の導線、商品の値付け、講座のカリキュラム、顧問契約のイメージ |
| デジタル | 予約フォーム、請求・見積、メールの下書き |
| 視覚 | フロー図、Before/After、1枚の設計図メタファー |

### Phase 4: 情報を絞る

- **今日の1テーマ**に集中し、脇道に広げすぎない
- **「今日の3つの要点」**を、上記表の①②③に沿って決める（テーマごとに文言を毎回書き直す）
- **全体の地図**は1テーマ用の短い3ブロック以内にする
- 細部・周辺知識は後半か折りたたみに回す
- **まとめ**に、成功の定義どおり **（1）一言（2）自分の事業と広い文脈（3）事業構築で押さえるポイント** が入る見込みか確認してから次へ進む

### Phase 5: 対話を設計

[references/dialogue-personas.md](references/dialogue-personas.md) に従い対話を設計する。Phase 6 で HTML に落とし込む。

### Phase 6: HTML 生成

1. **[references/exemplar-layout.html](references/exemplar-layout.html)** を開き、**セクションの並びと各ブロックのレイアウト**を見本に合わせる（導入1カード、地図グリッド、用語セクション、**h2「今日の3つの要点」**、本編の h2→対話2回→本文、まとめ、出典+免責、目次の `id` 対応）。**本編は1テーマについて3〜5セクション**まで増やしてよい（「依存」節の優先ルールに従い、論点は本スキル本节を正とする）。説明テキストは [references/exemplar.md](references/exemplar.md) の「黄金見本」。ブラウザ確認で公開版と見比べる場合は、[references/exemplar.md](references/exemplar.md) の「黄金見本」表の **公開 URL（ブラウザ確認用）** を参照（URL 文字列はその表にのみ書く。未設定ならローカルの exemplar-layout.html を開く）
2. [references/html-structure.md](references/html-structure.md) の**基本テンプレート**に従い HTML を組み立てる（CDN・フォント・スクリプトの入れ方を含む。テンプレート・配色・コンポーネント、表示・アイコン・装飾のルールは同ファイルの SSoT に従い、ここでは繰り返さない）
3. 用語解説ボックス（[references/term-dictionary.md](references/term-dictionary.md) の初出・定型・HTML例に従う）
4. コードや設定例を出す場合は、**必ず前に**「これがやること」を日本語で書く（書き方は [references/html-structure.md](references/html-structure.md) の「コード例の前の日本語」に合わせる）
5. 対話左枠は**毎回画像アバター**とする。画像・ファイル配置・デプロイ同梱は [references/html-structure.md](references/html-structure.md) の **「対話左枠の画像アバター」** に従う（中身は同見出しにのみ書く）。マークアップの型は [references/dialogue-personas.md](references/dialogue-personas.md) の **「対話左枠の画像アバター（必須）」** に従う。

### Phase 7: 批判的レビュー（サブエージェント）

**本スキル「読者・仕事・ゴール・『今日の3つの要点』」節の自分** の視点で忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "パーソナル図解の批判的レビュー",
  readonly: true,
  prompt: `
    レビュー視点は、本リポジトリの .claude/skills/diagram-personal/SKILL.md の「## 読者・仕事・ゴール・「今日の3つの要点」の意味」にある **読者・仕事・生活の文脈・成功の定義（ゴール）・まとめの必須3点** と一致させること。まずそのファイルを開き、該当見出し直下の箇条書きを前提としてから読み進めること（ペルソナをここに全文繰り返さない）。

    以下のHTML図解を読み、忖度なく批判してください：

    【HTMLファイルパス】{path}

    ## レビュー観点

    1. **まだ難しい言葉**: 説明があっても引っかかる語はどれか
    2. **たとえの違和感**: 混乱を増すたとえはないか
    3. **情報の鮮度と根拠**: 主張に対して、十分新しい出典があるか。日付のない古い記事だけで「現在」を語っていないか
    4. **情報過多**: 途中で離脱したくなる箇所。「今日の3つの要点」に絞れているか、**自分の事業への活かし方**がぼやけていないか
    5. **対話の不自然さ**: [references/dialogue-personas.md](references/dialogue-personas.md) のナビゲーター／学習者の方針に照らし、説教臭くないか、学習者が自分ごととして捉えられるか
    6. **構造・レイアウト**: [references/exemplar-layout.html](references/exemplar-layout.html) と比べ、セクション順・地図／用語／今日の3つの要点／本編の型・出典・目次が大きく崩れていないか（本編セクション数はスキル本节どおり3〜5でよい）
    7. **読後**: 今日の1テーマを30秒で説明できる自信があるか。**まとめの必須3点**（一言／自分の事業と広い文脈／事業構築のポイント）がはっきりしているか。**事業にどう活かすか**が一言で想像できるか（このスキル「成功の定義」と同じ基準）

    ## 出力形式

    ### 致命的な問題
    - ...

    ### 改善すべき点
    - ...

    ### 良かった点
    - ...

    ### 総評
    5段階評価: ⭐の数と一言
  `
})
```

### Phase 8: ブラッシュアップ

Phase 7 を反映して HTML を修正する。

**優先順位**:

1. 致命的な問題 → 必ず修正
2. 改善点 → 可能な範囲で対応
3. 良かった点 → 維持

**よくある修正**: 用語ボックス追加、たとえ差し替え、セクション分割、学習者セリフをより素朴に、「今日の3つの要点」の言い換え、**まとめ必須3点**の明確化、出典の日付表記、**自分用の一行・次の一歩**の明確化。

### Phase 9: デプロイ（必須）

**surge.sh へのデプロイは省略しない。** ローカル保存だけで完了としない。

1. デプロイ用に `index.html` を置いたディレクトリを用意する。対話用の **`images/`**（`learner.png` / `navigator.png`）を `index.html` と同じ階層に置く（[references/html-structure.md](references/html-structure.md) の「対話左枠の画像アバター」）
2. そのディレクトリで `surge . --domain {サブドメイン}.surge.sh` を実行する（`surge` 未インストールなら先にインストールする）
3. **公開 URL** をユーザーに伝える（下記の「公開 URL の決め方」で決めた **1 本だけ**を使う）
4. **手順 3 と同じ URL** をブラウザで開く（macOS: `open 'https://…'`、Linux: `xdg-open 'https://…'`、Windows: `start https://…` など）

**公開 URL の決め方（二重に決めない）**: `--domain foo.surge.sh` のように指定したときは **`https://foo.surge.sh/`** を正とする（チャットに書く URL も `open` に渡す URL もこれだけ）。ドメインを省略して surge に任せたときは、**ターミナルに surge が表示した URL** をそのまま正とする。別の URL を推測して手打ちしない。

必要なら併せて、リポジトリ内の適切なパスにも `index.html` を残してよい（SSoT は surge 上の **その 1 本の公開 URL** をユーザーに示し、同じ URL を開くこと）。
