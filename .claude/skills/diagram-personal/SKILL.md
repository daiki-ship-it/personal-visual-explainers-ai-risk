# Diagram Personal（パーソナル図解）

## 起動のしかた

**トリガーは「開始」だけ**（この文字列だけ、または「開始」が依頼の核になっている場合）。それ以外の入力では本スキルを起動しない。

ユーザーが「開始」したら **確認を挟まず** **Phase 0 から Phase 9 まで一気通貫**で進める。調査対象の競合はユーザーの指定ではなく **次の固定ミッション** に従う（下記「リサーチ対象の固定」）。

図解 HTML を生成し **surge.sh にデプロイする**（Phase 9 は省略しない）。

### リサーチ対象の固定（「開始」時の SSoT）

- **1回の「開始」＝競合分析は1主体だけ**（企業または個人のサービス提供者）。**複数社を1ページで並べ比べない**。
- **選定方式（方式A・固定）**: 実行エージェントが、**個人事業主向けのAIコンサル・AI顧問**（またはそれに極めて近い伴走・研修）を主たる収益の柱としている候補から **1件を選ぶ**。ユーザーから競合名・URLの入力は求めない。
- **既出主体との重複禁止**: 毎回の「開始」では、**これまでに本リポジトリで図解した競合主体**（企業・個人のサービス提供者）と**同一の主体を選ばない**。判定の根拠は **リポジトリ直下の `output/<各回>/index.html`**（`<title>`、主要見出し、導入文など、分析対象として明示されている**企業名・個人名・屋号・活動名**）。表記ゆれは同一主体とみなして除外する（例: 略称と正式名称が同じ主体を指す場合）。
- **`output/` が空のとき**: 過去成果がなければ重複チェックはスキップしてよい。
- **偏りについて**: 毎回の選定は検索結果・入手情報に依存するため、**網羅的な市場調査ではない**。図解の冒頭リード、または出典メモに **「今回は〇〇（観点：例・公開情報が揃っている／個人事業主向けの記述が明確、等）を選んだ」** と1行書いてよい。
- **調査で埋める観点**（図解で一目にする）:
  - **誰に**（ターゲット・ペルソナ）
  - **何を**（サービス名・提供内容）
  - **どのように**（オンライン／対面、伴走の度合い、期間）
  - **どの価格帯**（税込／税抜は分かる範囲で統一。不明なら「不明」と理由）
  - **自分のAI顧問・コンサル事業と重なる点・ずれる点**（勝ち筋の材料）

## 読者・仕事・ゴール・「この競合でまず押さえる3つ」の意味（本スキル内の正）

**読者・仕事・成功条件・競合1件の分析の解釈の正はこの節だけ**とする。他ファイルでは**同一内容を別の言い回しで長く書かない**（`exemplar.md` の `diagram-maji` 比較は**対照用の短い表**に限り、各セルは箇条書きレベルに留め、**読者・成功の定義本文は繰り返さず**本節や各リファレンスへのリンクに寄せる。チェックリストは本節へのリンクで足りる）。Phase 7 のレビュー用ペルソナは**この節と一致**させる（プロンプト内に全文を繰り返さず、実行時にこの節を読む前提にする）。

### 読者

- **読者は自分ひとり**。
- AI講師として活動はしているが、**IT業界全般への苦手意識が強く、IT業界での勤務経験はない**。
- **IT関連の専門用語はほぼわからず、抵抗感がある**。長文や用語の連続で疲れやすい。
- **ITリテラシーは低め**とみなし、誰が読んでも通じるレベルまで噛み砕く。

### 仕事・生活の文脈（セリフやたとえの軸）

- **いま**: 個人事業主として**AI講師**。集客から商品設計・セミナー登壇まで**自ら担当**。
- **今後（自分のAI顧問・コンサル事業の想定）**:
  - **ターゲット**: **個人事業主**。
  - **立ち位置**: 各クライアントの業務に**直接入り込み**、業務フローを再構築して **AI に最適化した業務形態**にする。**クライアントのAI部門として**入り込むイメージ。
  - **届け方の中心**: **オンラインセミナーが主役ではない**。**実際の業務フロー全体**を対象に最適化する。
  - **ツール**: 既存ツールの利用に加え、**個人に最適化した自動化ツールの開発**も行う。
  - **期間・価格（検討中）**: 伴奏はおおむね **3ヶ月**。価格は **15〜30万円**のレンジで検討中（図解・比較では税込／税抜は分かる範囲で統一。自分側の想定はこの帯）。

たとえ話は **講座・セミナー・クライアント対応・個人事業・業務フロー** になじむものを優先してよい（無理に入れない）。

### 典型ニーズ（優先度設計の参考）

ツールに競合分析を任せたが**文章が多すぎて理解不能**だった、という困りごとがある。**どの企業や個人が、誰に対して、どのような事業モデル・価格帯で、何を提供しているか**が**一目で**わかる図解を想定し、取捨選択する。**専門用語は日常語に翻訳**する。

### 成功の定義（ゴール）

読み終わったあとに次を満たすこと（レビュー・チェックリストもこの定義に揃える）。

1. **自分の言葉で、30秒程度に要点を説明できる**。「この競合は誰に・何を・どのくらいの価格で・どう届けているか」と、**自分が同じ市場でどこで勝つか**が腹落ちしている。
2. **競合1件の図解**ではさらに、**「この競合でまず押さえる3つ」**（下表）がそれぞれ一言で言え、**長文の調査レポートを読まなくてよい圧縮**になっていること。

### 「この競合でまず押さえる3つ」との対応

ページ上の **「この競合でまず押さえる3つ」** セクションの見出し文言はこれに固定する（旧称「まず覚える3つ」は使わない）。**本スキルは「開始」経路のみのため、題材は常に下表どおり（1競合の3要点）とする。**

| 題材 | 「この競合でまず押さえる3つ」の中身 |
|------|--------------------------------------|
| **競合1件の深掘り**（本スキル既定） | その競合について **ターゲット／提供内容・届け方／価格・市場での立ち位置** の3要点。各行に **一言の要約** と **自分のAI顧問・コンサル事業との関係（勝ち筋の材料）** を必ず入れる。 |

**全体の地図**は同一競合の **3つの側面**（上記3要点と対応）とする。**本編**は **1競合に対する深掘りを3セクション**に分ける（例: プロフィールと強み／商品と価格／あなたとの比較と勝ち筋）。`id` 名はトピックに合わせてよいが、**3セクション＝3側面と一貫**させる。

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**Phase 0 の前置（メインエージェント・本リポジトリ内）**

1. リポジトリ直下の **`output/`** を列挙し、各 **`output/*/index.html`** があれば読み、当該図解が扱っている**競合主体名**を抽出する（上記「リサーチ対象の固定」の **既出主体との重複禁止** に従う）。
2. 抽出結果を **「既出の競合（今回は選ばない）」一覧** にまとめる（該当がなければ **（なし）** とする）。

**explore サブエージェント**で、本スキル冒頭の **「リサーチ対象の固定」** に従い競合1件を調査する（候補を複数挙げたうえで **最終的に1件**に絞る）。`Task` を呼ぶときは、直下の `prompt` 内の **【既出の競合（今回は選ばない）】** に、前置手順2の一覧をそのまま貼り付けてから送る。

```
Task({
  subagent_type: "explore",
  description: "AIコンサル・顧問系 競合1件のリサーチ",
  prompt: `
    次のミッションで調査してください（対象はユーザー入力ではなく固定ルール）：

    【既出の競合（今回は選ばない）】
    （※呼び出し側が Phase 0 前置で作った一覧をここに列挙する。該当なしのときは「（なし）」）

    【ミッション】
    個人事業主向けの「AIコンサル」「AI顧問」またはそれに極めて近い伴走・研修を主商品とする事業者から、**分析対象をちょうど1件**選び、その競合について公開情報を集める。
    **【既出の競合（今回は選ばない）】に列挙された主体と、同一の企業・個人・屋号・活動と判断できる候補は採用しない**（表記が違っても同一なら除外）。
    除外の結果、妥当な候補が残らない場合は、検索キーワード・地域・紹介文の切り口を変え、**未掲載の別主体**を見つけるまで探索を続ける。
    選定は探索側の判断でよい（毎回偏りうること・網羅調査ではないことを理解したうえで、選んだ理由を1行メモする）。

    【調査項目】
    1. 候補の列挙（複数でよい）と、今回採用しなかった候補を外した簡単な理由
    2. **採用する1件**について：
       - 主体名（企業名または個人の活動名）
       - 誰に届けているか（ターゲット）
       - 何を提供しているか（商品・サービス名、内容の骨子）
       - どう届けているか（オンライン／対面、月次の伴走など）
       - 価格帯・課金の型（分からなければ不明とし、根拠を書く）
       - 強み・特徴（公開されている範囲）
       - 参照すべき一次・準一次ソースのURL（公式、料金ページ、募集ページ、信頼できるインタビュー記事など）
    3. **このユーザー（個人事業主のAI講師→今後AI顧問・コンサル）** にとっての示唆：重なる点・ずれる点（勝ち筋の仮説を1段落）
    4. 次回の「開始」で別の競合を調べたくなるメモがあれば1行

    結果は箇条書きで返してください。
  `
})
```

### Phase 1: ソース読み込み

1. Phase 0 で得た **採用1件**について、URL があれば `WebFetch` で取得し、足りなければ `WebSearch` 等で補う
2. **情報の全体像**（見出しの地図）を先に把握する
3. Phase 0 の「採用1件」と照合し、抜け・誤りを埋める

### Phase 2: 用語翻訳表を作る

技術用語・業界用語を洗い出し、日常語の名前とたとえを付ける。**変換の原則・初出・英語名の書き方・用語解説ボックスの定型・HTML例**はすべて [references/term-dictionary.md](references/term-dictionary.md) に従う（ここでは繰り返さない）。

### Phase 3: たとえ話を3つ以上

最終的なページ上のセクション順・塊の切り方は、[references/exemplar.md](references/exemplar.md) の「うまくいく構成」を SSoT とし、Phase 3〜6 でそこへ落とし込む。

| 種類 | 例 |
|-----|-----|
| 身近 | 予約台帳、レジ前の確認、教室のカリキュラム |
| デジタル | スマホの許可ダイアログ、メールの下書き |
| 視覚 | フロー図、地図、Before/After |

### Phase 4: 情報を絞る

- 細部・周辺知識は後半か折りたたみに回す
- **競合分析**では、スキル冒頭の成功定義どおり **3要点＋勝ち筋** に収まっているかを確認してから次へ進む

    以下のHTML図解を読み、忖度なく批判してください：

    6. **読後**: 自分の言葉で30秒程度に要点を説明できる自信があるか。競合のプロフィール・提供・価格がそれぞれ一言で言えるか。**どこで勝つか**が一言で言えるか（このスキル冒頭の「成功の定義」と同じ基準）

### Phase 8: ブラッシュアップ

Phase 7 を反映して HTML を修正する。

**優先順位**:

1. 致命的な問題 → 必ず修正
2. 改善点 → 可能な範囲で対応
3. 良かった点 → 維持

**よくある修正**: 用語ボックス追加、たとえ差し替え、セクション分割、学習者セリフをより素朴に、「この競合でまず押さえる3つ」の言い換え、勝ち筋の一文の明確化。

### Phase 9: デプロイ（必須）

**surge.sh へのデプロイは省略しない。** ローカル保存だけで完了としない。

1. デプロイ用に `index.html` を置いたディレクトリを用意する。対話用の **`images/`**（`learner.png` / `navigator.png`）を `index.html` と同じ階層に置く（[references/html-structure.md](references/html-structure.md) の「対話左枠の画像アバター」）
2. そのディレクトリで `surge . --domain {サブドメイン}.surge.sh` を実行する（`surge` 未インストールなら先にインストールする）
3. **公開 URL** をユーザーに伝える（下記の「公開 URL の決め方」で決めた **1 本だけ**を使う）
4. **手順 3 と同じ URL** をブラウザで開く（macOS: `open 'https://…'`、Linux: `xdg-open 'https://…'`、Windows: `start https://…` など）

**公開 URL の決め方（二重に決めない）**: `--domain foo.surge.sh` のように指定したときは **`https://foo.surge.sh/`** を正とする（チャットに書く URL も `open` に渡す URL もこれだけ）。ドメインを省略して surge に任せたときは、**ターミナルに surge が表示した URL** をそのまま正とする。別の URL を推測して手打ちしない。

必要なら併せて、リポジトリ内の適切なパスにも `index.html` を残してよい（SSoT は surge 上の **その 1 本の公開 URL** をユーザーに示し、同じ URL を開くこと）。
